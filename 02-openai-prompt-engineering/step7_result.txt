- GPUs may have different floating-point precision or math function behaviors.
- **Fix:**
  - Avoid relying on undefined floating point behavior.

- Ensure your workgroup sizes and ND-range are compatible with the Arch GPU limits.
- **Fix:** Query device info via SYCL runtime and tune your local/global ranges accordingly.

### 6. Dependency on CPU-only features or Host-side Data

When a SYCL kernel “works” on the CPU emulator but silently produces garbage on the GPU the three most common root causes are

  1. out-of-bounds indexing  
  2. missing synchronization (race) in local/shared memory  
  3. USM/buffer lifetime or placement mistakes  

On Intel Arc in particular the host‐emulation driver is very forgiving about races and out-of-bounds writes, so it “looks” correct, whereas the hardware happily scatters your bad data all over device memory.

Step 1: add bounds-checks and run under address-sanitizer  
– If your index is  
      auto gid = item.get_global_linear_id();  
  make sure gid < N.  
– If you launch a 2D or ND-range, double-check that you’re computing your flattening the same way in the kernel that you set up the ranges on the host.

Step 2: audit every use of local (shared) memory or atomics  
– If you do a per-work-group reduction into a `local_accessor‹T,1›`, you *must* synchronize before you read back from it. On the CPU driver you may never actually be hitting two threads in flight simultaneously, but on real hardware you absolutely are.  
  In SYCL-2020 you do this with either  
      it.barrier(access::fence_space::local_space);  
  or (if you’ve pulled in the `group<>` abstraction)  
      sycl::group_barrier(g, sycl::memory_scope::work_group);  
  *before* any thread reads or writes the shared array.  
– If you’re using `sycl::atomic_ref`, be sure you’ve picked the correct memory_scope. On-device you almost always want `memory_scope::device`, not `memory_scope::system`.

Step 3: verify your USM allocation or buffer/accessor usage  
– If you’re using USM, don’t allocate your device buffers with `malloc_host`. That only pins memory for CPU access; on Arc you either need `malloc_device` + an explicit `memcpy`, or—most conveniently—`malloc_shared`.  
– If you use buffers+accessors, make sure each `buffer` stays alive on the host until *after* the command‐group that writes it has completed.

Putting it all together, the “most likely” single fix is:

1) Switch your shared memory kernel to use a properly declared `local_accessor`  
2) Insert an explicit barrier after the last thread‐written store into it, e.g.

   ```cpp
   q.submit([&](sycl::handler &h) {
     local_accessor<float, 1> scratch{wg_size, h};
     h.parallel_for<class Foo>(
       nd_range<1>{global, wg_size},
       [=](nd_item<1> it) {
         // each work-item writes into scratch
         scratch[it.get_local_linear_id()] = …;
         // MAKE SURE you sync before reading
         it.barrier(access::fence_space::local_space);
         // now read-reduce from scratch[]
         …
       });
   });
   ```

That change alone will eliminate the race on local memory that the CPU driver accidentally masks and will almost certainly restore correctness on Arc. If you’re also using USM, change any `malloc_host` calls to `malloc_shared` (or switch entirely to buffers/accessors) and be sure to wait on your queue (or scope‐exit of the buffer) before touching host data.
